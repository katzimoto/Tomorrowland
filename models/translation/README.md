# Translation model bundle contract

This directory defines the **translation model bundle** contract — how Tomorrowland
packages, describes, and loads translation model files in both connected-build and
air-gapped runtime environments.

## Purpose

Tomorrowland's translation pipeline supports pluggable providers
(`TranslationProvider` ABC, #729). Many high-quality providers (OPUS-MT via
CTranslate2, NLLB-200) require model files that must be distributed alongside
the platform for offline/air-gapped operation.

This bundle format lets Tomorrowland:

- **Build** a self-contained `.tar.gz` bundle on a connected machine
  (download models, convert formats, checksum everything).
- **Transfer** the bundle to an air-gapped target host as a single file.
- **Validate** bundle integrity offline (checksum verification, manifest schema).
- **Load** model files into the target volume/path where the provider reads them.
- **Verify** at runtime that bundled models are present, complete, and match
  expected checksums so health checks can report readiness.

## Bundle layout

```
tomorrowland-translation-bundle-<provider>-<version>/
├── manifest.json           # Machine-readable bundle descriptor
├── checksums.txt           # SHA-256 checksums for every file in the bundle
├── README.md               # Human-readable notes (generated by build script)
└── models/                 # Provider-specific model file directory
    ├── <lang1>-<lang2>.<fmt>
    └── ...
```

- **`manifest.json`** follows the schema in `manifest.schema.json`.
- **`checksums.txt`** uses the standard `sha256sum` format (`<hash>  <path>`),
  relative to the bundle root.
- **`models/`** contains model files. The directory layout inside `models/` is
  provider-specific; the manifest `files` array records the exact relative path
  and checksum for every file.

## Manifest fields

| Field | Type | Description |
|---|---|---|
| `bundle_version` | string | Manifest format version (`"1.0"`). |
| `tomorrowland_release` | string | Tomorrowland release this bundle targets. |
| `created_at` | string | ISO-8601 UTC build timestamp. |
| `provider.name` | string | Provider key matching a `TranslationProvider` impl. |
| `provider.version` | string\|null | Detected provider version (e.g. `"libretranslate-1.6.3"`). |
| `provider.model_family` | string | Model family (`"argos"`, `"opus"`, `"nllb-200"`). |
| `provider.format` | string | Model file format (`"argos_package"`, `"ctranslate2"`, `"safetensors"`). |
| `supported_languages` | string[] | ISO 639-1 codes covered by the bundle. |
| `language_pairs` | object[] | Source→target pairs (`{"source":"en","target":"ar"}`). |
| `models_dir` | string | Relative path to model files (normally `"models"`). |
| `expected_env` | object | Recommended runtime env vars (e.g. `{"ARGOS_CHUNK_TYPE":"MINISBD"}`). |
| `files` | object[] | File inventory: `path`, `sha256`, optional `size_bytes`. |
| `license` | object | License metadata with verification status. |

## Runtime loading contract

When `TranslationProvider` implementations that require local model files are
initialized, they:

1. Read `manifest.json` from the configured bundle path.
2. Validate the manifest against `manifest.schema.json`.
3. Verify every file listed in the manifest exists and matches its checksum.
4. Report health status via the `health()` method — models present, checksums
   valid, language coverage.

If the bundle is missing or files fail checksum verification, the provider
must report `"status": "unhealthy"` in its health response with a descriptive
`"reason"` field so that operators can diagnose the issue offline.

## Build-time contract (connected environment)

The build script (`scripts/build-translation-model-bundle.sh`) is intended to
be run on a connected release host. It is responsible for:

1. Creating the bundle directory structure.
2. Downloading or generating model files (provider-specific).
3. Computing SHA-256 checksums for every model file.
4. Writing `manifest.json` populated with detected provider metadata.
5. Writing `checksums.txt`.
6. Creating the `.tar.gz` archive and `.sha256` outer checksum.

## Relationship to LibreTranslate Argos

The current bundled translation provider (LibreTranslate with Argos models)
**bakes its models into the Docker image** via `docker/libretranslate.Dockerfile`
and `docker/install-translation-packs.py`. The Argos packages reside inside the
`libretranslate_data` named volume at first container creation.

This bundle contract does **not** change how Argos models are distributed today.
It defines the contract for **future providers** (OPUS-MT, NLLB, etc.) that
will load model files from a local path at runtime.

When Tomorrowland adds a new provider that reads models from disk, it will:
- Implement `TranslationProvider` against this bundle contract.
- Accept a `bundle_path` pointing to an extracted `manifest.json`.
- Validate and load models from `bundle_path / models_dir` at startup.
